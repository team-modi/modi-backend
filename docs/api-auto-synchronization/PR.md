# API 자동 동기화 — 팀 가이드

> **한 줄 요약**: 백엔드가 API를 바꾸면, 프론트 레포에 "이렇게 맞춰야 합니다" PR이 자동으로 열리고 슬랙 알림이 옵니다. **여러분이 할 일은 그 PR을 리뷰하고 머지하는 것뿐**입니다. (자동 머지는 없습니다 — 승인은 항상 사람이 합니다.)

이 문서는 "API 자동 동기화 파이프라인"이 팀의 일상에서 어떻게 동작하는지 안내합니다. 백엔드/프론트 각각에서 무슨 일이 일어나는지, 새로 생긴 브랜치가 무엇인지, 그리고 **언제 동작하고 언제 안 하는지**를 다룹니다.

---

## 1. 한눈에

![OpenAPI Sync Pipeline 전체 구조](https://github.com/user-attachments/assets/57339939-3a47-4fdd-8704-044d0d64e7bf)

```
백엔드 develop에 API 변경 머지
  → (백엔드 CI) 앱 부팅·스펙 덤프·발행·신호 발사
  → (프론트 CI) 변경 감지·AI가 코드 수정·검증·PR 생성
  → 슬랙 알림
  → 프론트 개발자가 리뷰 & 머지  ← 사람이 하는 유일한 단계
```

전체 과정에서 **사람의 개입은 마지막 "리뷰 & 머지" 한 번뿐**입니다. 나머지는 전부 자동입니다.

---

## 2. 백엔드에서 일어나는 일 (`publish-openapi` 워크플로)

![백엔드 구간](https://github.com/user-attachments/assets/8cac2eae-bc0c-4128-b55c-bb4caa57f878)

백엔드 `develop`에 API 관련 변경이 머지되면 `publish-openapi` 워크플로가 자동으로 다음을 합니다.

| 단계 | 하는 일 |
|---|---|
| 1. 앱 부팅 | CI에서 실제 애플리케이션을 띄웁니다 (DB는 서비스 컨테이너, 시크릿은 GitHub Secrets로 주입) |
| 2. 스펙 덤프 | `/v3/api-docs`를 호출해 **현재 앱이 실제로 응답하는** OpenAPI 스펙을 받습니다 |
| 3. 정규화 | JSON 키를 정렬해서, 내용이 같으면 항상 같은 파일이 되도록 만듭니다 (헛 diff 방지) |
| 4. 발행 | 정규화된 `openapi.json`을 **`openapi-spec` 브랜치**에 커밋합니다 |
| 5. 신호 발사 | 스펙이 **실제로 바뀌었을 때만**, 프론트 레포로 `repository_dispatch`(신호)를 쏩니다. 이때 "누가 바꿨는지(작성자)"도 함께 실어 보냅니다 |

> **백엔드 개발자가 따로 할 일은 없습니다.** 평소처럼 API를 개발하고 develop에 머지하면, 나머지는 CI가 알아서 합니다. 슬랙 알림에 여러분 이름이 "백엔드 작업자"로 표시되는 이유는, 프론트가 궁금한 게 있을 때 누구에게 물어볼지 알 수 있게 하기 위함입니다.

## 3. 프론트에서 일어나는 일 (`api-sync` 워크플로)

![프론트 구간](https://github.com/user-attachments/assets/0498e727-2878-4498-9a97-09d2ea7bd7d4)

백엔드가 쏜 신호를 받으면 프론트 `api-sync` 워크플로가 다음을 합니다.

| 단계 | 하는 일 |
|---|---|
| 6. 변경 감지 | 새 스펙을 받아 직전 기준 스펙과 비교합니다. **다를 때만** 다음으로 진행합니다 |
| 7. 분류 (oasdiff) | 무엇이 어떻게 바뀌었는지 목록을 만들고, 하위호환을 깨는 변경(breaking)인지 분류합니다 |
| 8. API 자동 수정 (Claude Code) | 바뀐 계약에 맞게 `src/` 코드를 고칩니다. **API 클라이언트(`src/api`)뿐 아니라 그 데이터를 쓰는 컴포넌트·스토어(소비 지점)까지** 고칩니다. lint/build가 통과할 때까지 스스로 재시도합니다 |
| — 안전장치 | AI가 검사를 억지로 끄려 하면 폐기(anti-cheat), 허용 범위 밖을 건드리면 기계가 되돌림(가드레일) |
| 9. PR 생성 | 검증을 통과하면 develop 대상 PR을 만듭니다. 본문에 **변경 표 + "내 코드 영향" 분석**이 들어갑니다 |
| 10. 슬랙 알림 | 프론트 채널로 리뷰 요청 알림을 보냅니다 |

> **프론트 개발자가 할 일은 4번 항목 [팀원이 해야 할 일](#6-팀원이-실제로-해야-할-일)에서 정리합니다.**

PR 본문은 이렇게 생겼습니다 (실제 예시는 [7장](#7-실제-동작-화면) 참고):
- **⚡ 한눈에**: 이 PR이 코드를 고쳤는지 / 스펙만 갱신했는지 / 소비 지점까지 손댔는지(→ 반드시 리뷰)
- **📋 변경 내역**: 추가/변경/삭제 3섹션 표. 각 행에 `메서드·경로 / 무엇이 바뀜 / 내 코드 영향`
- 🤖 표시가 붙은 칸은 **AI의 추정**이니 반드시 실제 코드로 재확인
- 하단에 oasdiff 원본 changelog가 접혀 있음

## 4. 새로 생긴 브랜치 2개 — 건드리지 마세요

이 파이프라인 때문에 각 레포에 **자동 관리되는 브랜치**가 하나씩 생겼습니다. **직접 커밋하거나 삭제하지 마세요.** CI가 관리합니다.

| 레포 | 브랜치 | 무엇인가 |
|---|---|---|
| **modi-backend** | `openapi-spec` | 발행된 OpenAPI 스펙(계약서)만 담는 전용 브랜치. `develop`이 보호 브랜치라 직접 push가 안 돼서 만든 게시판 같은 공간입니다. 앱 코드와 섞이지 않고 "계약의 역사"만 남습니다. CI가 발행할 때마다 갱신됩니다 |
| **modi-frontend** | `api-sync/pending` | 자동 생성되는 sync PR이 올라가는 고정 브랜치. 실행마다 새 브랜치를 만들면 PR이 쌓이기 때문에, **열려 있는 sync PR 하나를 계속 갱신**하도록 고정 브랜치를 씁니다. 머지되면 다음 변경 때 다시 채워집니다 |

> 정리하면 — `openapi-spec`은 "백엔드가 발행한 최신 계약서", `api-sync/pending`은 "지금 리뷰 대기 중인 프론트 동기화 PR"입니다.

## 5. 언제 동작하고, 언제 동작 안 하나

가장 헷갈리기 쉬운 부분이라 명확히 정리합니다.

### ✅ 동작하는 경우

| 트리거 | 언제 |
|---|---|
| **백엔드 develop 머지** | `develop`에 push되고, 그 변경이 `src/main/**` 또는 `build.gradle*`를 건드렸을 때 (기본 경로) |
| **프론트 dispatch 수신** | 위 백엔드 발행이 신호를 쏘면 프론트가 즉시 기동 |
| **매일 스케줄** | 신호가 유실되는 경우를 대비해, 프론트가 하루 한 번 스스로 스펙을 확인 (백업 경로) |
| **수동 실행** | Actions 탭에서 `workflow_dispatch`로 직접 실행 (테스트·디버깅용) |

### ❌ 동작하지 않는(=PR이 안 생기는) 경우

| 상황 | 왜 |
|---|---|
| **develop이 아닌 브랜치**에 push | 백엔드 발행은 `develop` push에만 반응합니다. feature 브랜치 작업 중에는 안 돕니다 |
| develop 변경이 **`src/main`·`build.gradle`를 안 건드림** | 문서·테스트·설정만 바꾼 머지는 API 스펙과 무관하므로 발행하지 않습니다 (경로 필터) |
| API를 바꿨지만 **스펙 결과가 이전과 동일** | 예: 내부 로직만 리팩터링. 스펙이 바이트 단위로 같으면 "변경 없음"으로 판단하고 신호를 쏘지 않습니다 |
| 프론트가 받은 스펙이 **기준 스펙과 동일** | 이미 반영된 변경이면 프론트도 "변경 없음 - 종료"하고 PR을 만들지 않습니다 |
| **슬랙 알림만** 스킵되는 경우 | 슬랙 토큰이 없거나, PR에 실제 변화가 없는 재실행일 때는 알림만 조용히 건너뜁니다 (동기화 자체는 정상) |

> 핵심: **"API 스펙이 실제로 바뀌었을 때만"** 파이프라인이 끝까지 돕니다. 스펙에 영향 없는 변경은 조용히 아무 일도 하지 않습니다.

## 6. 팀원이 실제로 해야 할 일

### 백엔드 개발자

- **평소대로 개발하고 develop에 머지하면 끝입니다.** 별도 작업 없음.
- (참고) API를 바꿨는데 프론트에 sync PR이 안 떴다면 → [5장의 "동작하지 않는 경우"](#-동작하지-않는pr이-안-생기는-경우)를 확인하세요. 대개 스펙이 실제로는 안 바뀐 경우입니다.

### 프론트 개발자

1. 슬랙 알림(또는 프론트 레포의 열린 PR)을 확인합니다.
2. PR 본문의 **⚡ 한눈에**로 성격을 파악합니다.
   - "스펙만 갱신 · 코드 변경 없음" → 가볍게 확인 후 머지
   - "AI가 API 클라이언트 수정" → diff 확인 후 머지
   - "⚠️ 소비 지점 포함 — 반드시 리뷰" → **화면 코드가 바뀐 경우이니 꼼꼼히** 봅니다
3. **🤖 표시가 붙은 "내 코드 영향" 칸은 AI 추정**입니다. 그대로 믿지 말고 실제 코드·화면 동작을 확인하세요. (CI 검증은 lint+build까지만 하므로, 실제 UX 확인은 리뷰어의 몫입니다.)
4. 문제없으면 머지합니다. → develop에 반영 완료.

> **자동 머지는 없습니다.** AI가 만든 것은 어디까지나 "제안"이고, 그것이 제품으로서 옳은지는 사람이 판단합니다.

## 8. 자주 묻는 것

**Q. 백엔드 API를 바꿨는데 프론트에 PR이 안 떠요.**
→ 십중팔구 스펙이 실제로는 안 바뀐 경우입니다(내부 로직 변경 등). develop에 머지됐는지, `src/main` 변경이 있었는지, 그리고 발행된 스펙에 실제 차이가 있었는지 확인하세요. 정말 필요하면 백엔드 Actions 탭에서 `publish-openapi`를 수동 실행할 수 있습니다.

**Q. sync PR을 그냥 머지해도 되나요?**
→ 본문의 "한눈에"가 "스펙만 갱신"이면 안전합니다. 하지만 "소비 지점 포함"이거나 🤖 영향 칸에 실제 코드 변경이 적혀 있으면, 화면 동작을 확인하고 머지하세요. AI 추정을 그대로 신뢰하지 않는 것이 원칙입니다.

**Q. `openapi-spec` / `api-sync/pending` 브랜치를 지워도 되나요?**
→ 아니요. CI가 자동 관리하는 브랜치입니다. 지우면 파이프라인이 깨집니다.

**Q. 자동화를 잠깐 끄고 싶어요.**
→ 각 레포 Actions 탭에서 해당 워크플로(`publish-openapi`, `api-sync`)를 비활성화하면 됩니다.
